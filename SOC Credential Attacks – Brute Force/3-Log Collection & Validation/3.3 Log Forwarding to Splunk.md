### Architecture

The log forwarding process follows this flow:

Kali Linux (Attacker) → Windows 11 (Victim) → Splunk SIEM (Ubuntu Server)

Attack traffic generated from the attacker machine is logged on the Windows victim system.  
These logs are then forwarded to the Splunk server for indexing and analysis.

| Machine | Role | IP Address |
|--------|------|-----------|
| Kali Linux | Attacker | 192.168.1.106 |
| Windows 11 | Victim | 192.168.1.107 |
| Ubuntu Server | Splunk SIEM | 192.168.1.105 |

---

### Installing Splunk Universal Forwarder

1. Download **Splunk Universal Forwarder for Windows** from the official Splunk website.

2. Install the forwarder on the Windows victim machine.

3. During installation configure the **Splunk server IP**: 192.168.1.105

   
This allows the forwarder to send logs directly to the SIEM server.

---

### Configuring Log Inputs

The **inputs.conf** configuration file is used to define which logs should be forwarded to Splunk.

Example configuration:

```ini
[WinEventLog://Security]
index = windows
disabled = 0

[WinEventLog://Microsoft-Windows-Sysmon/Operational]
index = sysmon
disabled = 0
```
This configuration ensures that:

Windows Security logs are forwarded

Sysmon operational logs are forwarded

---

### Log Transmission

Once configured, the Universal Forwarder continuously sends logs from the Windows machine to the Splunk server.

These logs are then:

Indexed in Splunk

Searchable via SPL queries

Used to detect brute force attacks and suspicious authentication activity

---

### Verification

To confirm that logs are successfully received by Splunk, run the following search query:

```ini
index=windows OR index=sysmon
| stats count by sourcetype
```

If the configuration is correct, Splunk will begin displaying events from both Windows Security logs and Sysmon logs.
